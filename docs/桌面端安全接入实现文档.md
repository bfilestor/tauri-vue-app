# 桌面端安全接入实现文档

## 1. 文档目标

本文用于将“桌面端与后台管理端联动时如何保证数据交互安全、降低仿冒客户端接入风险”的需求，收敛为后续可实施的服务端安全接入方案。

本文以以下文档为依据：

- `doc/产品需求说明书.md`
- `doc/产品功能开发说明书.md`
- `doc/服务端接口设计.md`
- `doc/数据库表设计.md`
- `AGENTS.md`

本文定位为实现文档，不替代详细代码设计，也不替代第三方安全合规评审。

## 2. 问题定义

本项目中的“联动”指的是：

- 桌面端用户通过 `/app-api/**` 完成登录、上传、OCR、AI 分析、下单、查询
- 后台管理人员通过 `/admin-api/**` 完成运营、补偿、风控、审计、查询
- 两端共享同一套订单、钱包、AI、风控业务数据，但访问入口、用户体系、权限体系严格隔离

本方案要解决的核心风险包括：

1. 其他人仿照桌面端协议，自行编写脚本或壳客户端直接接入 `/app-api/**`
2. 抓包后重放高价值请求，重复触发 AI 调用、查询或下单
3. 被盗 token 在陌生设备上复用
4. 桌面端误用或恶意访问 `/admin-api/**`
5. 异常高频、异常地域、异常版本客户端持续消耗资源且无法快速定位

必须明确：

- 桌面端软件天然可被逆向，无法通过单一技术手段彻底杜绝仿冒
- 目标不是“绝对无法仿冒”，而是“提高仿冒成本、降低批量滥用成功率、提升识别与处置速度”

## 3. 设计原则

### 3.1 通道隔离优先

- 桌面端只允许访问 `/app-api/**`
- 后台管理端只允许访问 `/admin-api/**`
- `/admin-api/**` 继续沿用 RuoYi 后台登录、权限码、操作日志体系
- `/app-api/**` 使用独立业务用户登录态，禁止与后台 `sys_user` 混用

### 3.2 用户鉴权与客户端鉴别分层

- 用户鉴权回答“当前是谁在操作”
- 客户端鉴别回答“当前请求是否来自已登记、未冻结、未吊销的受控客户端”
- 两者必须同时成立，高价值接口才能放行

### 3.3 高价值接口强校验，普通接口可渐进收口

优先强制校验的接口：

- 登录完成后的资料上传相关接口
- OCR / AI 分析 / 智能问答接口
- 下单、支付结果查询、退款结果查询
- 钱包余额、台账、导出、批量查询

可先以灰度方式接入的接口：

- 资料类只读查询
- 账户中心普通展示查询

### 3.4 安全链统一承接

- 不在业务 Controller 中手工解析 token、时间戳、签名
- 客户端签名、防重放、设备状态校验优先沉淀到 `qcy-framework`
- 设备、审计、风控、冻结、吊销等业务能力沉淀到 `qcy-health`

## 4. 总体方案

## 4.1 安全架构分层

1. 传输层安全
   - 服务端强制 HTTPS
   - 桌面端建议启用服务端证书固定或公钥固定
2. 用户登录态
   - 继续复用现有 token 安全链识别业务用户
3. 客户端身份鉴别
   - 设备首次激活登记
   - 为设备签发受控凭证
   - 后续请求携带签名头
4. 防重放
   - 服务端校验时间窗、随机串唯一性、摘要一致性
5. 风控与审计
   - 对签名失败、版本异常、设备冻结、异常频率进行拦截与记录
6. 后台运营
   - 后台可查看设备状态、审计日志、失败原因，并执行冻结/解冻/吊销

## 4.2 请求判定模型

服务端在 `/app-api/**` 上的高价值接口应按以下顺序判定：

1. 是否命中匿名白名单
2. 是否存在合法业务用户登录态
3. 是否存在合法 `clientId/deviceId`
4. 是否在允许时间窗内
5. `nonce` 是否已被使用
6. 请求摘要与签名是否匹配
7. 设备状态是否为可用
8. 风控规则是否放行

任一失败即拒绝请求，并记录审计事件。

## 5. 设备激活与凭证模型

## 5.1 设备标识

桌面端接入时统一携带：

- `clientId`：客户端发行标识，用于区分发行渠道或客户端产品线
- `deviceId`：设备唯一标识
- `appVersion`：客户端版本号
- `deviceType / osName / osVersion`：基础设备信息

建议设备状态机：

- `PENDING`：已发起激活，待确认
- `ACTIVE`：已激活，可正常访问
- `FROZEN`：已冻结，拒绝访问
- `REVOKED`：已吊销，不再信任
- `EXPIRED`：凭证过期，需要重新激活

现有 `hg_user_device` 已具备设备追踪基础能力，建议在后续实现中补齐受控接入所需字段，避免另起一套重复设备主表。

## 5.2 凭证模式

推荐主方案：

- 桌面端首次激活时本地生成设备公私钥
- 服务端保存设备公钥和凭证版本
- 每次请求由客户端使用私钥签名
- 服务端使用公钥验签

过渡方案：

- 设备激活后由服务端下发 `deviceSecret`
- 客户端按约定报文进行 HMAC 签名
- 后续再升级为公私钥方案

推荐优先级：

1. 生产长期方案采用“设备私钥签名 + 服务端公钥验签”
2. 若首期交付时间紧张，可先落地 HMAC 版，但必须同时预留密钥轮换和设备冻结能力

## 5.3 激活流程

建议最小激活流程如下：

1. 桌面端安装后生成本地设备信息和设备密钥
2. 客户端调用激活申请接口，提交 `clientId/deviceId/appVersion/deviceFingerprint/publicKey`
3. 服务端写入设备记录并返回激活结果
4. 若采用人工审核或邀请码策略，则先进入 `PENDING`
5. 激活成功后设备状态改为 `ACTIVE`
6. 后续登录、刷新 token、AI、下单等接口均要求带设备签名头

若源文档暂未要求人工审核，则首期采用“自动激活 + 后台可冻结/吊销”的更窄实现。

### 5.3.1 激活请求字段（首期定稿）

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| `clientId` | 是 | 客户端发行标识，如 `desktop-tauri` |
| `deviceId` | 是 | 设备唯一标识，客户端稳定生成 |
| `deviceName` | 否 | 设备名称，便于后台识别 |
| `deviceType` | 是 | 设备类型，如 `PC / MAC` |
| `osName` | 是 | 操作系统名称 |
| `osVersion` | 是 | 操作系统版本 |
| `appVersion` | 是 | 当前客户端版本 |
| `deviceFingerprint` | 是 | 客户端按约定生成的设备指纹摘要 |
| `credentialMode` | 是 | `ASYMMETRIC / HMAC` |
| `publicKey` | 条件必填 | `ASYMMETRIC` 模式必填，提交设备公钥 |
| `secretProof` | 条件必填 | `HMAC` 过渡模式下必填，证明已持有初始化密钥材料 |
| `activateTime` | 是 | 客户端发起激活时间 |
| `nonce` | 是 | 激活请求随机串，避免重复提交 |
| `signature` | 是 | 对激活 canonical string 的签名结果 |

### 5.3.2 激活响应字段（首期定稿）

| 字段 | 是否返回 | 说明 |
| --- | --- | --- |
| `activationStatus` | 是 | `PENDING / ACTIVE` |
| `deviceStatus` | 是 | 当前设备状态 |
| `credentialVersion` | 是 | 凭证版本号，后续轮换依据 |
| `serverTime` | 是 | 服务端当前时间 |
| `expireTime` | 否 | 凭证过期时间，`ACTIVE` 时建议返回 |
| `deviceSecret` | 条件返回 | 仅 HMAC 过渡方案返回，且必须提示客户端安全存储 |
| `message` | 是 | 激活结果描述 |

### 5.3.3 激活错误码（首期定稿）

| 错误码 | 含义 | 说明 |
| --- | --- | --- |
| `CLIENT_DEVICE_INVALID` | 客户端标识非法 | `clientId/deviceId` 缺失或格式非法 |
| `CLIENT_SIGNATURE_INVALID` | 激活签名错误 | 签名不匹配或签名材料错误 |
| `CLIENT_TIMESTAMP_EXPIRED` | 激活时间窗超限 | 请求过期或时间漂移过大 |
| `CLIENT_NONCE_REPLAY` | 激活请求重放 | 相同 `nonce` 在窗口内重复使用 |
| `CLIENT_DEVICE_FROZEN` | 设备已冻结 | 不允许继续激活或续期 |
| `CLIENT_DEVICE_REVOKED` | 设备已吊销 | 必须重新登记或人工处理 |
| `CLIENT_VERSION_BLOCKED` | 客户端版本不允许 | 命中版本门禁或灰度策略 |

### 5.3.4 设备状态迁移规则（首期定稿）

| 当前状态 | 触发条件 | 下一状态 | 备注 |
| --- | --- | --- | --- |
| `PENDING` | 自动激活策略放行 | `ACTIVE` | 首期默认走该路径 |
| `PENDING` | 人工拒绝/风险命中 | `REVOKED` | 人工审批不是首期必做，先保留口径 |
| `ACTIVE` | 后台冻结 / 自动风控冻结 | `FROZEN` | 禁止访问高价值接口 |
| `ACTIVE` | 后台吊销 / 密钥失效 | `REVOKED` | 必须重新登记 |
| `ACTIVE` | 过期未续期 | `EXPIRED` | 允许走续期或重激活 |
| `FROZEN` | 后台解冻 | `ACTIVE` | 恢复访问 |
| `EXPIRED` | 续期成功 | `ACTIVE` | 更新过期时间 |
| `REVOKED` | 重新发起激活且审核通过 | `PENDING/ACTIVE` | 取决于激活策略 |

明确收敛结论：

- 首期不展开人工审批流，只记录 `PENDING` 口径并默认自动转 `ACTIVE`
- 首期若无法立即上公私钥，允许 HMAC 过渡，但必须保留 `credentialVersion`、冻结、吊销、续期能力
- 高价值接口命中 `FROZEN / REVOKED / EXPIRED` 均直接拒绝，禁止业务层自行放过

## 6. 请求签名协议

## 6.1 请求头约定

桌面端已激活设备访问 `/app-api/**` 时，建议统一增加：

- `X-Client-Id`
- `X-Device-Id`
- `X-Timestamp`
- `X-Nonce`
- `X-Signature`
- `X-App-Version`

业务 token 仍通过：

- `Authorization: Bearer {accessToken}`

## 6.2 签名原文

建议采用固定 canonical string，避免各端实现不一致：

```text
HTTP_METHOD \n
REQUEST_PATH \n
BODY_SHA256 \n
X-Timestamp \n
X-Nonce \n
X-Client-Id \n
X-Device-Id
```

说明：

- `REQUEST_PATH` 仅包含服务端实际路由路径与排序后的 query string
- `BODY_SHA256` 对空 body 也要计算固定摘要
- 时间戳建议统一使用毫秒或秒级 UTC 时间，并在文档中固定
- 同一端必须始终使用同一编码、换行和排序规则
- 首期 HMAC 过渡方案按 `HmacSHA256` 后再做 Base64 输出，避免不同语言默认 hex/base64 口径不一致

## 6.3 验签规则

- 服务端按 `clientId + deviceId` 找到有效设备记录
- 校验设备状态是否允许访问
- 校验时间窗，建议首期控制在 `300s`
- 校验 `nonce` 是否在窗口内已使用
- 使用设备公钥或 `deviceSecret` 校验 `X-Signature`
- 校验通过后再进入业务鉴权和业务处理

## 6.4 防重放

建议组合使用以下机制：

- 时间窗校验：过旧或未来时间直接拒绝
- `nonce` 一次性校验：使用 Redis 以 `deviceId + nonce` 作为短 TTL key
- 幂等键：AI 调用、下单等场景继续保留 `Idempotency-Key`

首期建议：

- `nonce` TTL 与签名时间窗一致
- 重放命中时直接返回统一错误码
- 风控侧同步记录重放次数与命中设备

## 7. 模块落点

遵循 `AGENTS.md` 约束，建议模块归属如下：

### 7.1 `qcy-framework`

负责通用安全链能力：

- 客户端签名过滤器或拦截器
- 签名 canonical 组装器
- 验签组件
- 时间窗与 `nonce` 防重放基础校验
- 统一安全上下文注入
- 白名单与灰度开关配置

### 7.2 `qcy-health.auth`

负责前台登录与设备登录态联动：

- 登录成功时补登记设备信息
- 刷新 token 时校验设备状态
- 退出登录时更新最近会话状态

### 7.3 `qcy-health.user`

负责设备主体资料与用户绑定关系：

- 设备注册、更新、查询
- 设备状态迁移
- 设备基础信息与最后活跃时间维护

### 7.4 `qcy-health.risk`

负责风控和审计：

- 签名失败审计
- 重放拦截审计
- 异常设备、异常 IP、异常地域、异常版本识别
- 冻结、解冻、吊销编排

### 7.5 `qcy-admin`

负责接口入口，不承载复杂逻辑：

- `/app-api/client/**` 激活类接口
- `/admin-api/health/risk/**` 安全审计查询接口
- `/admin-api/health/user/**` 设备管理接口

## 8. 接口设计建议

## 8.1 前台接口

建议新增最小接口：

- `POST /app-api/client/activate`
  - 设备激活申请
- `POST /app-api/client/refresh-credential`
  - 设备凭证轮换或续期
- `GET /app-api/account/devices`
  - 已有接口继续作为个人设备查询入口

说明：

- 当前业务登录接口可保持不变，但在成功登录后应把设备信息与设备状态联动起来
- 若首期不新增独立激活接口，也至少需要在登录成功时完成设备登记和状态校验

## 8.2 后台接口

建议新增最小接口：

- `GET /admin-api/health/user/devices`
  - 分页查询设备
- `GET /admin-api/health/user/devices/{deviceId}`
  - 查询设备详情和最近审计
- `PUT /admin-api/health/user/devices/{deviceId}/freeze`
  - 冻结设备
- `PUT /admin-api/health/user/devices/{deviceId}/unfreeze`
  - 解冻设备
- `PUT /admin-api/health/user/devices/{deviceId}/revoke`
  - 吊销设备
- `GET /admin-api/health/risk/client-audits`
  - 查询签名失败、重放、异常访问审计

后台接口默认必须：

- 使用 `@PreAuthorize`
- 关键写操作使用 `@Log`
- 返回 `AjaxResult` 或 `TableDataInfo`

## 9. 数据与存储建议

## 9.1 设备主数据

优先复用 `hg_user_device` 作为设备主体表，并补充以下能力字段：

- `client_id`
- `credential_version`
- `public_key` 或 `secret_digest`
- `activate_time`
- `expire_time`
- `last_signature_time`
- `last_signature_fail_time`
- `sign_fail_count`
- `status_reason`

## 9.2 防重放缓存

建议使用 Redis 保存短周期 `nonce`：

- key：`health:client:nonce:{deviceId}:{nonce}`
- value：固定占位
- TTL：与签名时间窗一致

## 9.3 审计数据

建议通过风险审计能力统一沉淀：

- 签名失败
- 时间窗超限
- `nonce` 重放
- 设备冻结命中
- 非法访问后台接口
- 版本不在允许范围

首期可以先写入现有风控/审计能力的扩展表或日志表，避免为了安全能力单独再造一整套日志体系。

## 10. 风控处置规则

建议最小处置策略如下：

1. 单设备短时间内连续签名失败超过阈值时，进入人工观察
2. 单设备短时间内连续命中重放时，自动冻结
3. 已冻结设备访问任何高价值接口时直接拒绝
4. 低版本或非允许版本客户端仅允许访问升级提示相关接口
5. 同一账号在异常设备快速切换时，触发二次校验或限制高价值操作

首期不要把风控做成复杂评分平台，优先做规则化、可审计、可人工干预的窄实现。

## 11. 实施顺序建议

建议按以下顺序落地：

### 第一阶段：协议与设备底座

- 明确签名协议
- 完成设备字段扩展与状态机
- 完成激活/登记能力

### 第二阶段：服务端验签闭环

- 在 `qcy-framework` 接入签名过滤器
- 接入时间窗、`nonce` 防重放
- 建立统一错误码和审计日志

### 第三阶段：高价值接口强校验

- 对 AI、订单、钱包、导出等接口启用强校验
- 对普通查询接口保持灰度开关

### 第四阶段：后台运营与风控

- 提供设备管理和审计查询
- 支持冻结、解冻、吊销
- 接入版本门禁和异常策略

## 12. 桌面端接入清单（给客户端团队）

桌面端团队在接入时，至少需要完成以下事项：

1. 生成并稳定保存 `clientId / deviceId`
2. 在本地安全存储 `deviceSecret` 或设备私钥
3. 所有高价值 `/app-api/**` 请求统一增加：
   - `X-Client-Id`
   - `X-Device-Id`
   - `X-Timestamp`
   - `X-Nonce`
   - `X-Signature`
   - `X-App-Version`
4. 按 canonical string 规则组装待签名原文
5. 对 `CLIENT_VERSION_BLOCKED / CLIENT_SIGNATURE_INVALID / CLIENT_NONCE_REPLAY` 做明确提示
6. 在收到 `FROZEN / REVOKED / EXPIRED` 类错误时，提示用户重新激活或联系管理员

桌面端建议补充但首期不强制的安全项：

- 服务端证书固定 / 公钥固定
- 设备私钥存操作系统安全区（Keychain / Credential Manager / Secret Service）
- 对日志中的敏感头做脱敏，不打印完整签名材料

## 13. 灰度上线步骤（首期可执行方案）

### 13.1 上线顺序

1. 先发布服务端，但保留普通查询灰度放行
2. 桌面端新版本接入签名头、版本号、激活流程
3. 先仅对 AI、订单、钱包、usage 开启强校验
4. 观察 24-72 小时的签名失败率、版本阻断率、nonce 重放命中率
5. 指标稳定后，再评估是否收紧账户中心和资料类查询接口

### 13.2 观察指标

建议重点观察：

- `CLIENT_SIGNATURE_INVALID` 次数
- `CLIENT_NONCE_REPLAY` 次数
- `CLIENT_TIMESTAMP_EXPIRED` 次数
- `CLIENT_VERSION_BLOCKED` 次数
- 同设备连续失败次数是否快速升高
- 冻结/吊销后是否仍有访问命中

### 13.3 回滚策略

若出现大面积误伤或旧客户端无法使用：

1. 将 `health.client-security.strict-mode=false`
2. 保持 `strict-paths` 不变，但允许 `gray-paths` 按配置继续放行
3. 如需进一步止损，可临时将 `enabled=false`，整体关闭客户端安全过滤器
4. 保留 Redis 审计计数与设备状态，不做数据回滚，便于事后排查

### 13.4 兼容窗口建议

- 普通查询接口保留 1-2 个版本周期灰度窗口
- 高价值接口默认直接启用强校验
- 若确有历史老客户端存量，可仅放开查询，不放开 AI、订单、钱包写操作

## 14. 源文档缺口与本次收敛结论

当前源文档仍存在以下缺口：

- 未明确桌面端激活是否需要人工审核或邀请码
- 未明确设备指纹算法与 Tauri 客户端具体存储方式
- 未明确是否必须支持多发行渠道、多环境独立 `clientId`
- 未明确审计日志最终是否复用现有表还是新增专表

基于“优先更窄实现”的原则，本次实现文档给出如下收敛结论：

1. 首期不设计桌面端直连后台能力
2. 首期以“业务 token + 设备签名 + 时间窗 + nonce + 风控审计”构成最小闭环
3. 首期优先支持自动激活、后台冻结/吊销、Redis 防重放
4. 首期优先保护 AI、订单、钱包等高价值接口，其余接口灰度接入
5. 若时间受限，可先落地 HMAC 版签名，再演进为公私钥版

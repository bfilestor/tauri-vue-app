# 医疗健康资料 OCR 与 AI 智能分析系统

## 1. 文档信息

- 文档名称：服务端接口设计
- 适用版本：V2.0
- 服务端技术架构：Spring Boot + RuoYi-Vue
- 接口使用方：Tauri 桌面端、RuoYi-Vue 后台管理端
- 设计目标：定义用户体系、家庭成员、商品订单、支付、次数钱包、内置 AI 网关相关接口，作为服务端开发与客户端联调依据

## 2. 设计原则

- 服务端只管理账号、成员、订单、支付、钱包、调用流水，不默认保存完整医疗图片与分析正文
- 所有收费逻辑在服务端完成，客户端只发起请求，不决定扣费结果
- 所有内置 AI 请求必须具备鉴权、幂等、审计和失败返还能力
- 用户自定义 AI 调用不经过本服务端，不在本接口范围内
- 接口统一采用 JSON，文件上传场景采用 `multipart/form-data`

## 3. 基础约定

## 3.1 接口前缀

- 前台客户端接口前缀：`/app-api`
- 后台管理接口前缀：`/admin-api`
- 支付回调接口前缀：`/open-api`

## 3.2 鉴权方式

- 登录成功后返回 `accessToken` 与 `refreshToken`
- 需要登录的接口统一在请求头传递：`Authorization: Bearer {accessToken}`
- 刷新令牌接口单独使用 `refreshToken`

## 3.3 通用响应结构

建议沿用 RuoYi 风格并适配前台接口，统一返回：

```json
{
  "code": 200,
  "msg": "操作成功",
  "data": {},
  "traceId": "7b7a4d6d0d624c20ab2d0f6d7f56e991"
}
```

字段说明：

- `code`：业务状态码，`200` 表示成功
- `msg`：提示信息
- `data`：业务数据
- `traceId`：链路追踪 ID，便于排障

## 3.4 分页结构

```json
{
  "code": 200,
  "msg": "操作成功",
  "data": {
    "list": [],
    "total": 100,
    "pageNum": 1,
    "pageSize": 20,
    "hasMore": true
  },
  "traceId": "..."
}
```

## 3.5 常用业务状态码

| code | 含义 |
|------|------|
| 200 | 成功 |
| 400 | 参数错误 |
| 401 | 未登录或令牌失效 |
| 403 | 无权限或账号被限制 |
| 404 | 数据不存在 |
| 409 | 幂等冲突、状态冲突 |
| 410 | 次数余额不足 |
| 429 | 请求过于频繁 |
| 500 | 服务异常 |
| 502 | 第三方 AI 或支付通道异常 |

## 10. 内置 AI 网关接口

## 10.1 通用要求

- 所有网关接口必须登录
- 所有请求必须传 `requestId`，用于幂等控制
- 所有请求必须传 `memberId`，用于成员级隔离和调用审计
- 服务端默认不落盘原图和完整分析正文
- 网关内部负责调用第三方模型服务、扣费和返还

## 10.2 内置 OCR 接口

- 方法：`POST`
- 路径：`/app-api/builtin-ai/ocr`
- Content-Type：`multipart/form-data`

表单字段：

- `requestId`：请求幂等 ID
- `memberId`：家庭成员 ID
- `file`：单张图片文件
- `fileName`：原始文件名
- `mimeType`：文件类型
- `bizType`：默认 `ocr`

响应示例：

```json
{
  "code": 200,
  "msg": "识别成功",
  "data": {
    "requestId": "ocr-20260327110001",
    "sceneType": "ocr",
    "chargeStatus": "success",
    "chargedCredits": 1,
    "remainingBalance": 47,
    "modelName": "builtin-ocr-v1",
    "result": {
      "rawText": "...",
      "parsedItems": [
        {
          "name": "血红蛋白",
          "value": "120",
          "unit": "g/L",
          "referenceRange": "115-150",
          "isAbnormal": false
        }
      ]
    }
  },
  "traceId": "..."
}
```

说明：

- 服务端返回结果后，客户端将 OCR 结果落本地库
- 失败时按策略返回 `chargeStatus=refund` 或 `none`

## 12. 幂等与事务要求

## 12.1 幂等要求

- 登录验证码发送按手机号 + 场景限流
- 创建订单按 `clientRequestId` 幂等
- 内置 AI 调用按 `requestId` 幂等
- 支付回调按三方交易号 + 回调事件幂等

## 12.2 事务要求

以下操作必须放入事务：

- 订单支付成功 + 钱包加次 + 流水落账
- AI 调用成功/失败 + 钱包扣减/返还 + 使用记录更新
- 人工补偿 + 钱包变更 + 台账记录

## 13. 安全与风控要求

- 验证码接口需限频与图形验证码保护
- 登录接口需记录设备与 IP
- 网关接口需校验账号状态、设备状态、风控状态
- 对超频用户做限流或熔断
- 对异常高频调用写审计日志
- 对日志中的手机号、token、支付敏感字段做脱敏

## 14. 客户端联调建议

- 客户端登录后先调用 `profile`、`family-members`、`wallet` 三个接口初始化用户态
- 家庭成员切换仅切换本地上下文，不要求服务端长期保存“当前成员”状态
- 内置 AI 调用前先可选调用 `usage/precheck` 做前置提示
- 正式扣费以 `builtin-ai/*` 接口返回结果为准
- 所有 `requestId` 建议由客户端本地生成并保证唯一

## 16. 结论

该接口设计以“本地隐私数据不上传、云端负责账号与商业化闭环”为核心，适配 Spring Boot + RuoYi-Vue 的业务模块化实现方式。前台接口服务于 Tauri 客户端登录、成员管理、充值购买和内置 AI 调用；后台接口服务于商品、订单、钱包、调用日志和风控运营，能够作为后续详细表结构设计和服务端开发落地的直接基础。

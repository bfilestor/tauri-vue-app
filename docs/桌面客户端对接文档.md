# 桌面客户端对接文档

## 1. 文档说明

- 文档目标：为 Tauri + Vue3 桌面客户端提供一份可直接参考的服务端对接说明，覆盖安全接入、接口清单、调用顺序和示例代码。
- 整理依据：
  - `doc/桌面端安全接入实现文档.md`
  - `doc/服务端接口设计.md`
  - `doc/前端对接API清单.md`
  - `doc/前端对接API示例.md`
- 适用对象：
  - 桌面客户端开发
  - Java 版联调工具、测试工具、后台集成程序
  - Vue3 / Tauri 客户端接入层开发

本文优先站在“桌面客户端如何接服务端”的视角进行收敛。

需要特别说明：

1. 桌面端只能访问 `/app-api/**`，不能直接访问 `/admin-api/**`
2. 设备安全头用于“客户端身份鉴别”，不能替代业务用户 `accessToken`
3. 高价值接口应同时满足“用户已登录 + 设备已激活 + 请求签名正确”
4. 当前仓库同时存在“设计口径”和“当前 MVP 实现口径”，正式联调时：
   - 协议设计、未来落地方向以本文和 `doc/服务端接口设计.md` 为准
   - 如需对接当前已存在的演示接口，可同时参考 `doc/前端对接API清单.md`

## 2. 接入总览

### 2.1 通道说明

| 通道 | 前缀 | 桌面端是否可调用 | 说明 |
| --- | --- | --- | --- |
| App 业务接口 | `/app-api/**` | 是 | 桌面端业务调用统一走这里 |
| 后台管理接口 | `/admin-api/**` | 否 | 仅后台管理端使用 |
| 开放回调接口 | `/open-api/**` | 否 | 仅第三方平台回调使用 |

### 2.2 推荐接入顺序

1. 客户端生成并保存 `clientId`、`deviceId`
2. 首次安装后调用设备激活接口
3. 激活成功后保存设备凭证
4. 用户进行登录，获得 `accessToken` / `refreshToken`
5. 后续调用资料、商品、订单、钱包、AI 等 `/app-api/**` 接口
6. 高价值接口统一追加设备签名头
7. `accessToken` 过期后调用刷新令牌接口
8. 凭证过期或服务端要求轮换时调用设备凭证续期接口

### 2.3 推荐的高价值接口范围

以下接口建议默认强制校验签名头：

- `/app-api/orders/**`
- `/app-api/wallet/**`
- `/app-api/usage/**`
- `/app-api/builtin-ai/**`
- `/app-api/account/balance`
- `/app-api/account/ledger`

以下接口可按灰度策略逐步收口：

- `/app-api/account/profile`
- `/app-api/family-members/**`
- `/app-api/products/**`

## 3. 统一请求约定

### 3.1 统一请求头

已登录接口：

```http
Authorization: Bearer {accessToken}
Content-Type: application/json
X-Trace-Id: {traceId}
```

已激活设备的桌面端请求，额外建议携带：

```http
X-Client-Id: desktop-tauri
X-Device-Id: win10-7c1d9a8f
X-Timestamp: 1775000000000
X-Nonce: 7bdfecf7c7a24e3db81f4f72b32b6f78
X-Signature: Base64Signature
X-App-Version: 1.0.0
```

AI、下单等幂等场景建议追加：

```http
Idempotency-Key: ocr-20260401-0001
```

### 3.2 统一返回格式

普通接口：

```json
{
  "code": 200,
  "msg": "操作成功",
  "data": {}
}
```

分页接口：

```json
{
  "code": 200,
  "msg": "查询成功",
  "rows": [],
  "total": 0
}
```

### 3.3 统一签名原文

推荐 canonical string：

```text
HTTP_METHOD \n
REQUEST_PATH \n
BODY_SHA256 \n
X-Timestamp \n
X-Nonce \n
X-Client-Id \n
X-Device-Id
```

说明：

- `HTTP_METHOD` 使用大写，如 `POST`
- `REQUEST_PATH` 包含路径和排序后的 query string
- `BODY_SHA256` 对空 body 也要计算固定摘要
- `X-Timestamp` 建议统一使用毫秒级 UTC 时间戳
- `X-Signature` 首期可使用 `HmacSHA256 + Base64`
- 长期建议升级为“设备私钥签名 + 服务端公钥验签”

### 3.4 统一错误处理建议

客户端至少要识别以下错误码或错误语义：

| 错误码 | 含义 | 客户端处理建议 |
| --- | --- | --- |
| `CLIENT_SIGNATURE_INVALID` | 签名不合法 | 提示安全校验失败，必要时重新激活 |
| `CLIENT_TIMESTAMP_EXPIRED` | 请求时间窗超限 | 自动校准时间后重试一次 |
| `CLIENT_NONCE_REPLAY` | 请求疑似重放 | 重新生成 `nonce` 与幂等键后重试 |
| `CLIENT_DEVICE_FROZEN` | 设备被冻结 | 提示联系管理员 |
| `CLIENT_DEVICE_REVOKED` | 设备已吊销 | 重新走设备激活 |
| `CLIENT_VERSION_BLOCKED` | 当前版本被限制 | 提示用户升级客户端 |
| 业务余额不足 | 钱包次数不足 | 引导用户购买或联系运营 |

## 4. 推荐调用流程

### 4.1 首次安装

1. 客户端生成 `deviceId`
2. 客户端生成设备密钥或准备 HMAC 初始材料
3. 调用 `POST /app-api/client/activate`
4. 保存 `credentialVersion`、`expireTime` 以及 `deviceSecret` 或私钥

### 4.2 日常登录

1. 调用发送验证码或邮箱验证码接口
2. 调用登录或注册接口
3. 保存 `accessToken`、`refreshToken`、`userInfo`
4. 后续所有登录态接口带 `Authorization`

### 4.3 AI 使用流程

1. 查询成员列表，确定 `memberId`
2. 查询钱包或调用使用预检接口
3. 发起 OCR / 分析 / 对话请求
4. 记录返回的 `usageNo`、`traceId`
5. 如失败且返回可重试错误，再决定是否重试

### 4.4 下单支付流程

1. 查询商品列表与 SKU
2. 创建订单
3. 获取支付二维码
4. 轮询订单状态
5. 支付成功后刷新钱包余额

## 5. 可用接口说明

本节优先列出桌面端应对接的 `/app-api/**` 接口，不展开后台接口与第三方回调接口。

### 5.1 设备安全接入接口

#### 5.1.1 POST `/app-api/client/activate`

- 说明：设备首次激活申请
- 是否登录：否
- 是否需要签名：是

关键请求字段：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `clientId` | 是 | 客户端发行标识，如 `desktop-tauri` |
| `deviceId` | 是 | 设备唯一标识 |
| `deviceName` | 否 | 设备名称 |
| `deviceType` | 是 | `PC` / `MAC` |
| `osName` | 是 | 操作系统名称 |
| `osVersion` | 是 | 操作系统版本 |
| `appVersion` | 是 | 客户端版本号 |
| `deviceFingerprint` | 是 | 设备指纹摘要 |
| `credentialMode` | 是 | `ASYMMETRIC` 或 `HMAC` |
| `publicKey` | 条件必填 | 非对称模式传设备公钥 |
| `secretProof` | 条件必填 | HMAC 过渡模式下的证明材料 |
| `activateTime` | 是 | 激活时间 |
| `nonce` | 是 | 请求随机串 |
| `signature` | 是 | 激活签名 |

关键响应字段：

| 字段 | 说明 |
| --- | --- |
| `activationStatus` | `PENDING` / `ACTIVE` |
| `deviceStatus` | 当前设备状态 |
| `credentialVersion` | 凭证版本 |
| `serverTime` | 服务端时间 |
| `expireTime` | 过期时间 |
| `deviceSecret` | 仅 HMAC 模式返回 |
| `message` | 描述信息 |

#### 5.1.2 POST `/app-api/client/refresh-credential`

- 说明：设备凭证轮换、续期
- 是否登录：建议是
- 是否需要签名：是

建议请求字段：

- `clientId`
- `deviceId`
- `credentialVersion`
- `appVersion`
- `nonce`
- `timestamp`

## 7. Java 对接示例

以下示例使用 Java 17 + `java.net.http.HttpClient`，演示 HMAC 过渡模式。

### 7.1 生成签名头

```java
import com.fasterxml.jackson.databind.ObjectMapper;

import javax.crypto.Mac;
import javax.crypto.spec.SecretKeySpec;
import java.net.URI;
import java.net.http.HttpClient;
import java.net.http.HttpRequest;
import java.net.http.HttpResponse;
import java.nio.charset.StandardCharsets;
import java.security.MessageDigest;
import java.time.Instant;
import java.util.Base64;
import java.util.Map;
import java.util.UUID;

public class HealthAppClient {

    private static final ObjectMapper MAPPER = new ObjectMapper();
    private final HttpClient httpClient = HttpClient.newHttpClient();

    private final String baseUrl;
    private final String clientId;
    private final String deviceId;
    private final String appVersion;
    private final String deviceSecret;
    private String accessToken;

    public HealthAppClient(String baseUrl, String clientId, String deviceId, String appVersion, String deviceSecret) {
        this.baseUrl = baseUrl;
        this.clientId = clientId;
        this.deviceId = deviceId;
        this.appVersion = appVersion;
        this.deviceSecret = deviceSecret;
    }

    public void setAccessToken(String accessToken) {
        this.accessToken = accessToken;
    }

    private String sha256Base64(String content) throws Exception {
        MessageDigest digest = MessageDigest.getInstance("SHA-256");
        byte[] hash = digest.digest(content.getBytes(StandardCharsets.UTF_8));
        return Base64.getEncoder().encodeToString(hash);
    }

    private String sign(String method, String requestPath, String body, String timestamp, String nonce) throws Exception {
        String bodySha256 = sha256Base64(body == null ? "" : body);
        String canonical = String.join("\n",
                method.toUpperCase(),
                requestPath,
                bodySha256,
                timestamp,
                nonce,
                clientId,
                deviceId
        );
        Mac mac = Mac.getInstance("HmacSHA256");
        mac.init(new SecretKeySpec(deviceSecret.getBytes(StandardCharsets.UTF_8), "HmacSHA256"));
        byte[] signBytes = mac.doFinal(canonical.getBytes(StandardCharsets.UTF_8));
        return Base64.getEncoder().encodeToString(signBytes);
    }

    public String postJson(String path, Object bodyObj, boolean withAuth) throws Exception {
        String body = MAPPER.writeValueAsString(bodyObj);
        String timestamp = String.valueOf(Instant.now().toEpochMilli());
        String nonce = UUID.randomUUID().toString().replace("-", "");
        String signature = sign("POST", path, body, timestamp, nonce);

        HttpRequest.Builder builder = HttpRequest.newBuilder()
                .uri(URI.create(baseUrl + path))
                .header("Content-Type", "application/json")
                .header("X-Client-Id", clientId)
                .header("X-Device-Id", deviceId)
                .header("X-Timestamp", timestamp)
                .header("X-Nonce", nonce)
                .header("X-Signature", signature)
                .header("X-App-Version", appVersion)
                .POST(HttpRequest.BodyPublishers.ofString(body));

        if (withAuth && accessToken != null && !accessToken.isBlank()) {
            builder.header("Authorization", "Bearer " + accessToken);
        }

        HttpResponse<String> response = httpClient.send(builder.build(), HttpResponse.BodyHandlers.ofString());
        return response.body();
    }

    public static void main(String[] args) throws Exception {
        HealthAppClient client = new HealthAppClient(
                "https://api.example.com",
                "desktop-tauri",
                "win10-7c1d9a8f",
                "1.0.0",
                "your-device-secret"
        );

        String loginResp = client.postJson("/app-api/auth/login/password", Map.of(
                "userName", "demoUser",
                "password", "Demo123456",
                "deviceId", "win10-7c1d9a8f",
                "deviceName", "Windows Desktop"
        ), false);

        System.out.println(loginResp);
    }
}
```

- 生产环境不要把 `deviceSecret` 明文写死到源码中
- 如果是桌面 Java 客户端，优先使用系统安全存储
- 若后续升级为非对称签名，只需替换 `sign()` 实现，调用层可以保持不变

以下示例适用于 Vue3 + Axios。若项目基于 Tauri，建议把密钥保存、签名计算优先下沉到 Tauri Rust 侧或系统安全区，而不是长期保存在 `localStorage`。

### 8.1 `src/utils/clientSecurity.ts`

```ts
import CryptoJS from 'crypto-js'

export interface ClientSecurityContext {
  clientId: string
  deviceId: string
  appVersion: string
  deviceSecret: string
}

export function sha256Base64(raw: string) {
  const hash = CryptoJS.SHA256(CryptoJS.enc.Utf8.parse(raw))
  return CryptoJS.enc.Base64.stringify(hash)
}

export function buildCanonicalString(
  method: string,
  requestPath: string,
  body: string,
  timestamp: string,
  nonce: string,
  clientId: string,
  deviceId: string
) {
  return [
    method.toUpperCase(),
    requestPath,
    sha256Base64(body || ''),
    timestamp,
    nonce,
    clientId,
    deviceId
  ].join('\n')
}

export function signRequest(
  ctx: ClientSecurityContext,
  method: string,
  requestPath: string,
  body: string,
  timestamp: string,
  nonce: string
) {
  const canonical = buildCanonicalString(
    method,
    requestPath,
    body,
    timestamp,
    nonce,
    ctx.clientId,
    ctx.deviceId
  )
  const sign = CryptoJS.HmacSHA256(canonical, ctx.deviceSecret)
  return CryptoJS.enc.Base64.stringify(sign)
}

export function createNonce() {
  return crypto.randomUUID().replace(/-/g, '')
}
```

### 8.2 `src/utils/request.ts`

```ts
import axios from 'axios'
import { createNonce, signRequest, type ClientSecurityContext } from './clientSecurity'

const request = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL,
  timeout: 15000
})

function getSecurityContext(): ClientSecurityContext | null {
  const raw = localStorage.getItem('health-client-security')
  return raw ? JSON.parse(raw) : null
}

function getAccessToken() {
  return localStorage.getItem('health-access-token') || ''
}

request.interceptors.request.use((config) => {
  const token = getAccessToken()
  if (token) {
    config.headers.Authorization = `Bearer ${token}`
  }

  const security = getSecurityContext()
  if (!security) {
    return config
  }

  const method = (config.method || 'get').toUpperCase()
  const url = config.url || ''
  const queryString = config.params
    ? '?' + new URLSearchParams(config.params as Record<string, string>).toString()
    : ''
  const requestPath = `${url}${queryString}`
  const body = config.data ? JSON.stringify(config.data) : ''
  const timestamp = Date.now().toString()
  const nonce = createNonce()
  const signature = signRequest(security, method, requestPath, body, timestamp, nonce)

  config.headers['X-Client-Id'] = security.clientId
  config.headers['X-Device-Id'] = security.deviceId
  config.headers['X-Timestamp'] = timestamp
  config.headers['X-Nonce'] = nonce
  config.headers['X-Signature'] = signature
  config.headers['X-App-Version'] = security.appVersion

  return config
})

export default request
```

## 9. 联调建议

### 9.1 客户端最少要做的事情

1. 固定生成并稳定保存 `clientId`、`deviceId`
2. 完成设备激活与凭证保存
3. 所有高价值请求统一加签名头
4. 登录态和设备凭证分开管理
5. AI、订单类请求统一生成 `Idempotency-Key`
6. 对冻结、吊销、版本阻断、签名失败做明确提示

### 9.2 不建议的做法

- 不要让桌面端直接访问 `/admin-api/**`
- 不要把 `deviceSecret` 长期明文散落在前端页面代码
- 不要由页面组件自行拼装签名头，应该统一在请求层完成
- 不要把 `accessToken` 当作设备校验凭证使用

### 9.3 联调排查顺序

如果请求失败，建议按以下顺序排查：

1. `Authorization` 是否过期
2. `X-Client-Id` / `X-Device-Id` 是否正确
3. `X-Timestamp` 是否超出时间窗
4. `X-Nonce` 是否重复
5. `REQUEST_PATH` 与 query string 排序是否一致
6. `BODY_SHA256` 是否和真实 body 一致
7. `Idempotency-Key` 是否误复用

## 10. 与当前 MVP 实现的口径差异说明

当前仓库内已有一批可联调的 MVP 接口，和本文中的设计口径存在少量差异，主要包括：

1. 部分当前 `/app-api/**` 接口仍显式传 `userId`
2. 部分列表接口当前返回 `AjaxResult.data` 数组，而不是标准分页结构
3. 部分 AI、支付、验证码流程仍是 Mock 实现
4. 设备激活与签名校验属于当前规划中的安全增强能力，联调时需确认服务端是否已开启

因此建议：

- 新一版桌面客户端封装优先按本文的“目标对接口径”设计
- 若当前要立即联调现有仓库，再额外对照 `doc/前端对接API清单.md` 做一次适配

## 11. 一句话接入结论

桌面客户端接服务端时，应始终遵循“只走 `/app-api/**`、登录态与设备签名并行、AI/订单/钱包等高价值接口统一加签、统一由请求层封装安全头和幂等键”的原则，这样才能同时兼顾当前联调效率和后续正式上线的安全要求。

# Epic E1 · 认证与会话基础

## 1. Epic 概览

- **Epic ID：** E1
- **目标：** 为桌面端补齐设备激活、认证接入、令牌管理和登录注册访客交互基础能力。
- **业务价值：** 没有稳定的身份与会话层，就无法承接购买次数包、账户余额展示和高价值 AI 接口安全校验。
- **范围内：** `clientId/deviceId` 持久化、设备激活、签名请求层、认证 API 封装、登录注册访客弹框、会话同步和退出登录。
- **范围外：** 服务端接口实现、后台管理端、微信扫码登录、账号资料编辑详情页。
- **上游依赖：** `docs/桌面客户端对接文档.md`
- **账户管理服务端信息** 地址 http://192.168.1.75/   测试用户 用户名：zhaoyifan，手机号：13800000005 ，密码：   a12345678
- **风险/源文档缺口：** 缺少正式开发说明书；设计稿偏手机号口径，而接口文档同时存在邮箱注册与密码登录；本 Epic 采用最窄实现口径：支持设备激活、密码登录、注册送 1 次试用、临时访客入口，并为验证码登录保留扩展位。

## 2. Story 列表

### Story E1-S1 · 设备接入与认证请求基础

**用户故事：** 作为桌面端用户，我希望客户端能安全保存设备身份并统一处理认证请求，以便后续登录、支付和 AI 调用都能稳定联调。

**验收摘要：**
- AC1: 首次启动后客户端能够生成并持久化 `clientId`、`deviceId`，并完成设备激活。
- AC2: 高价值接口通过统一请求层附加签名头、令牌头和幂等键。

#### Issue E1-S1-I1 · 本地设备凭证与签名请求层

- **目标：** 建立桌面端统一的安全请求基础设施，负责设备标识、激活凭证和签名头拼装。
- **范围：** 本地安全存储键定义、激活流程封装、签名 canonical string、请求层自动注入 `X-Client-Id/X-Device-Id/X-Timestamp/X-Nonce/X-Signature/X-App-Version`。
- **依赖：** 无
- **备注/风险：** 若服务端尚未开启设备激活或签名校验，需要保持可灰度关闭的兼容开关。

##### 验收标准
- AC1: 首次启动若无本地设备凭证，自动生成并尝试激活设备。
- AC2: 激活成功后持久化 `deviceSecret/privateKey`、`credentialVersion`、过期时间。
- AC3: 统一请求层可按接口类型附加签名头与幂等键，页面组件不直接拼签名。

| FT-1 | 首次启动自动生成设备标识 | 本地无客户端安全存储 | 启动应用并初始化请求层 | 生成并保存 `clientId`、`deviceId`，触发一次激活请求 |
| FT-2 | 高价值接口自动附加签名头 | 已存在有效设备凭证 | 发起订单或 usage 预检请求 | 请求头包含签名相关字段，且页面代码无需重复拼装 |

| 编号 | 用例名称 | 前置条件 | 输入/操作 | 预期结果 |

| BT-1 | 设备凭证过期自动续期 | 本地存在过期 `credentialExpireTime` | 发起高价值请求 | 先执行续期或重新激活，再继续请求 |
| BT-2 | 服务端未开启签名校验 | 服务端返回设备激活未开通或签名不要求 | 初始化请求层 | 客户端记录兼容状态，不因签名缺失阻断低价值接口 |

#### Issue E1-S1-I2 · 认证 API 与会话存储封装

- **目标：** 提供认证相关 API 层和会话状态容器，统一管理登录、注册、刷新令牌、登出和访客态。
- **范围：** `/app-api/auth/*` API 封装、`accessToken/refreshToken/userInfo` 持久化、401 自动刷新、访客态标识。
- **依赖：** E1-S1-I1
- **备注/风险：** 注册字段与登录字段需采用最窄口径；若验证码接口未联通，需要保留错误提示而非伪造流程。

##### 验收标准
- AC1: 登录成功后持久化令牌与用户信息，并可恢复会话。
- AC2: 注册成功后直接进入已登录状态，并可标记“赠送 1 次试用”提示。
- AC3: 令牌过期时请求层可自动刷新一次，失败后回退到未登录态。

### Story E1-S2 · 登录注册访客交互与应用壳同步

**用户故事：** 作为终端用户，我希望在桌面端看到清晰的登录、注册和访客入口，并让侧边栏用户区随会话状态即时变化，以便快速完成身份切换。

**验收摘要：**
- AC1: 未登录时展示登录/注册入口，支持临时访客。
- AC2: 登录后展示头像、昵称、剩余次数和账号菜单入口。

#### Issue E1-S2-I1 · 登录/注册/临时访客弹框

- **目标：** 按设计稿实现认证弹框，包括登录、注册和临时访客三种入口。
- **范围：** 弹框结构、Tab 切换、表单校验、注册送试用提示、服务条款/隐私政策占位链接、调用认证 API。
- **依赖：** E1-S1-I2
- **备注/风险：** `登陆弹框.png` 与 `注册弹框.png` 未提供结构化尺寸标注，UI 以 `stitch/sidebar-user-area.html` 与 `stitch/model-service.html` 的共有交互为准。

##### 验收标准
- AC1: 弹框默认提供登录和注册切换，并包含临时访客入口。
- AC2: 登录表单至少校验账号和密码非空；注册表单至少校验账号、验证码、密码和确认动作。
- AC3: 注册面板明确展示“注册即送 1 次免费 AI 调用体验”。

#### Issue E1-S2-I2 · 侧边栏用户区与会话状态联动

- **目标：** 按侧边栏设计稿实现用户区，并和登录态、访客态、余额数据联动。
- **范围：** Layout 侧边栏底部用户区域、登录按钮、注册按钮、已登录卡片、账号菜单、退出入口、会话恢复。
- **依赖：** E1-S1-I2、E1-S2-I1
- **备注/风险：** 当前 Layout 底部仍为静态“健康用户/本地管理”；需要替换为动态渲染，但不改现有主导航信息架构。

##### 验收标准
- AC1: 未登录时显示登录/注册按钮与注册送试用提示。
- AC2: 已登录时显示昵称、在线状态、剩余次数进度和快捷入口。
- AC3: 重启应用后可根据本地会话恢复用户区展示。
